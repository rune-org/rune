# AGENTS.md

Guidance for AI coding agents working in the Rune repository.

> [!IMPORTANT]
> Keep this file current. If you add a service, change test/lint/build workflows, move generated-code boundaries, or alter cross-service message/API contracts, update this file in the same change.

## Working Principles

These rules apply across the repository:

- **Make surgical changes**. Every changed line should trace back to the user request.
- Prefer **simple, readable code** over speculative abstractions.
- **Match the local style** of the package you are editing, even when another style would also work.
- **State assumptions when behavior is ambiguous**; ask only when a safe local decision is not possible.
- **Verify changes with the narrowest meaningful command**, then broaden when the change crosses service boundaries.
- **Do not edit generated artifacts** by hand. Update their source and regenerate.
- **Do not clean up unrelated files**, formatting, or dead code unless explicitly asked.

## Overview

Rune is a workflow automation platform composed of the following microservices:

- `apps/web/`: Next.js + React frontend + React Flow for the visual workflow canvas.
- `services/api/`: FastAPI backend for auth, users, workflows, credentials, permissions, templates, Smith/Scryb, OAuth, webhooks, and internal service endpoints.
- `services/rune-worker/`: Go workflow execution engine. Consumes RabbitMQ execution messages, executes DSL nodes, publishes node status and completion events.
- `services/rtes/`: Rust real-time execution service. Stores execution state/history and serves HTTP/WebSocket execution updates.
- `services/archivist/`: Python completion recorder. Consumes workflow completion events and updates PostgreSQL execution rows.
- `services/scheduler/`: Python polling scheduler. Finds due scheduled workflows and calls the API internal run endpoint.
- `dsl/`: Language-neutral workflow DSL definition and generator for TypeScript, Python, and Go types.

All build, run, test, format, and lint commands are orchestrated via the root `Makefile`. Please refer to the root [README.md](README.md) for the complete list of available root-level commands.


## Generated Code Boundaries

- `apps/web/src/client/` is generated by HEY-API from the backend OpenAPI schema. Never edit it directly. Change FastAPI models/routes, regenerate the client using the project workflow, then consume it through wrappers in `apps/web/src/lib/api/`.
- `dsl/generated/types.ts`, `dsl/generated/types.py`, and `dsl/generated/types.go` are generated from `dsl/dsl-definition.json`. Never edit generated DSL files directly. Update `dsl/dsl-definition.json`, run `make dsl-generate`, then run `make dsl-test`.

## Cross-Service Contracts

Treat these as shared contracts, not implementation details:

- Workflow definitions are represented by the DSL in `dsl/dsl-definition.json` and consumed by the frontend, API, and worker.
- Workflow run messages are published by `services/api/src/workflow/queue.py` and consumed by the Go worker. The payload shape is `NodeExecutionMessage` in `services/api/src/workflow/schemas.py`.
- RabbitMQ workflow queues/routing keys are shared across services. Go constants live in `services/rune-worker/pkg/platform/queue/constants.go`; RTES and archivist bind to the same `workflows` topic exchange.
- RTES authorization depends on execution tokens published by the API to the token queue. Keep token payload fields compatible with `services/rtes/src/domain/models.rs`.
- Archivist keeps a minimal copy of the API execution model in `services/archivist/src/models.py`. If the API `Execution` table or status enum changes, update archivist in the same change.
- Scheduler calls `POST /internal/workflows/{workflow_id}/run` with `X-Internal-Key`. Changes to internal auth or response shape must update `services/scheduler/src/client.py`.

## Frontend (`apps/web/`)

- Stack: Next.js, React, TypeScript strict, Tailwind, Radix UI/shadcn-style components, and Vitest.
- Commands:
  ```bash
  cd apps/web
  pnpm dev
  pnpm build
  pnpm lint
  pnpm format
  pnpm typecheck
  pnpm test
  pnpm test:coverage
  pnpm format:check
  ```
- `src/client/` is generated by HEY-API from the backend OpenAPI schema. Never modify it by hand. To change API types or operations, update FastAPI models/routes and regenerate the client through the project workflow.
- Use `@/*` imports for `src/*`.
- Use `cn()` from `@/lib/cn` for class merging.
- Reuse `src/components/ui/`, `src/components/`, and existing feature patterns before creating new UI.
- UI components in `src/components/ui/` should stay presentational. Put domain behavior in `src/features/` or `src/lib/`.
- Feature components live in `src/features/`; the canvas feature is the densest area and contains its own components, context, hooks, nodes, edges, stores, styles, types, and utilities.
- Layout components live in `src/components/layout/`; auth, credentials, dashboard, workflows, admin, shared, and home components live under their matching `src/components/*` directories.
- Use `lucide-react` for icons.
- Fonts are Noto Sans for body text, Playfair Display for display text, and Cinzel for brand text.
- Use API wrappers in `src/lib/api/` instead of importing generated SDK operations directly.
- Route groups: `(app)` protected app routes, `(home)` public auth/setup routes, `(canvas)` workflow canvas routes.
- Forms use React Hook Form + Zod.
- State management uses React Context + `useReducer`; do not introduce Redux/Zustand unless explicitly requested and justified.
- Important contexts include auth state in `@/lib/auth`, app/user/workflow state in `@/lib/state`, and canvas execution/graph state under `@/features/canvas/context/`.
- Tests use Vitest, jsdom, and React Testing Library.
- Colocate tests as `*.test.ts` or `*.test.tsx` beside the source under test.
- Import RTL helpers from `@/test/render`, not `@testing-library/react` directly.
- Do not hit the real network from frontend tests. Mock API wrappers from `@/lib/api/*`.

## API (`services/api/`)

FastAPI service using Python 3.13, `uv`, SQLModel/async SQLAlchemy, Redis, RabbitMQ, Alembic, and LangGraph/LiteLLM integrations.

```bash
cd services/api
uv sync
uv run fastapi dev src/app.py --reload-dir src
uv run pytest
uv run ruff check src/ test/
uv run ruff format src/ test/
```

Repository conventions:

- Use absolute imports from `src.*`.
- Routers are registered in `src/app.py`; keep feature routers, schemas, dependencies, and services colocated by domain.
- Keep request/response validation in Pydantic schemas. Keep database persistence in services/repositories, not routers.
- Use shared exception/response patterns from `src/core/`.
- Always add/adjust Alembic migrations for any database model changes:
  ```bash
  make db-revision msg="describe change"
  make db-upgrade
  ```
- Tests live under `services/api/test/` and mirror `src/` domains. Use existing async fixtures from `test/conftest.py`.
- Do not bypass permission helpers for workflows, credentials, users, or internal endpoints.
- When API schema changes affect the frontend, update wrappers and regenerated client artifacts through the established generation path.

## Worker (`services/rune-worker/`)

Go execution engine for workflow DSL nodes.

```bash
cd services/rune-worker
go mod download
go run cmd/worker/main.go
go test ./...
go vet ./...
gofmt -w .
```

Use the worker test runner for integration/E2E suites:

```bash
cd services/rune-worker
./scripts/run_tests.py unit
./scripts/run_tests.py integration --start-services
./scripts/run_tests.py e2e --start-services
./scripts/run_tests.py all --start-services -v
```

Conventions:

- If adding or changing a node type, update the DSL definition, generated DSL types, frontend canvas behavior, worker implementation, and tests together.
- On adding or changing nodes, make sure to update the relevant RFC under `services/rune-worker/rfcs/`.
- Read the relevant RFC under `services/rune-worker/rfcs/` before changing established node semantics such as switch, split/aggregate, merge, wait, edit, list transform, datetime, or integration nodes.

## RTES (`services/rtes/`)

Rust real-time execution service.

```bash
cd services/rtes
cargo run
cargo test
cargo fmt
cargo clippy --all-targets --all-features
```

Conventions:

- HTTP/WebSocket routes live under `src/api/`.
- Domain message models live in `src/domain/models.rs`.
- MongoDB execution persistence lives in `src/infra/execution_store.rs`.
- RabbitMQ consumers live in `src/infra/messaging.rs`.
- Redis execution-token storage lives in `src/infra/token_store.rs`.
- The WebSocket stream loads persisted execution state before live updates. Preserve that behavior when changing history/live streaming.
- Current limitation: split node executions are documented as unsupported/corrupting in RTES. Do not claim support without implementing and testing it end to end.
- The crate enables strict lints in `Cargo.toml`; avoid `unwrap`, `expect`, `panic`, `dbg!`, and stdout/stderr prints in production paths.

## Archivist (`services/archivist/`)

Python service that records completion messages into PostgreSQL.

```bash
cd services/archivist
uv sync
uv run python -m src.main
```

Conventions:

- Consumes `workflow.completion` from the durable `workflows` topic exchange.
- Keep `src/models.py` synchronized with the API execution table/status fields.
- Keep this service intentionally small; do not import API internals just to share models.

## Scheduler (`services/scheduler/`)

Python service that polls due scheduled workflows and triggers API internal runs.

```bash
cd services/scheduler
uv sync
uv run python -m src.main
```

Conventions:

- Polling SQL lives in `src/db.py`; API calls live in `src/client.py`.
- Advances `next_run_at` before making API calls. Preserve or explicitly revisit that behavior if changing retry semantics.

When adding a node type:

- Add it to `dsl/dsl-definition.json`.
- Regenerate TypeScript/Python/Go types.
- Add worker execution behavior and tests.
- Update `nodeRegistry.ts` for canvas node metadata.
- Update frontend canvas node configuration, credential filtering, validation, and UI as needed.
- Update API validation/storage behavior if the saved workflow shape changes.

## Database And Migrations

- API database models live in `services/api/src/db/models.py`.
- Alembic migrations live in `services/api/migrations/`.
- Use root Make targets for migrations:
  ```bash
  make db-revision msg="short description"
  make db-upgrade
  make db-current
  make db-history
  ```
- `make db-reset` is destructive. Do not run it unless the user explicitly asks for a reset.

## Testing Expectations

Pick tests based on blast radius:

- Frontend-only change: `cd apps/web && pnpm test`, plus `pnpm typecheck` and/or `pnpm lint` for TypeScript/UI changes.
- API-only change: `make test-infra-up`, then `make api-test` or targeted `cd services/api && uv run pytest test/path`.
- Worker unit change: `cd services/rune-worker && go test ./...`.
- Worker queue/infrastructure change: use `./scripts/run_tests.py integration --start-services`.
- DSL change: `make dsl-test`.
- Cross-service workflow execution change: test API queue publishing, worker execution, RTES/archivist consumers, and frontend assumptions where applicable.

If a verification command cannot run because dependencies or infrastructure are unavailable, report that explicitly with the command you attempted.
