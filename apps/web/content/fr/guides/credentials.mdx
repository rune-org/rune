# Gérer les identifiants

## Identifiants OAuth2 (apportez votre propre client OAuth)

Rune utilise le flux **authorization code** : votre navigateur visite le fournisseur, vous approuvez l'accès, puis l'**API** échange le `code` à usage unique contre `access_token` / `refresh_token` à l'aide de votre **client secret** (le navigateur ne voit jamais le secret).

### URI de redirection à enregistrer

Dans les paramètres du client OAuth de votre fournisseur (par exemple Google Cloud Console → APIs & Services → Credentials → OAuth 2.0 Client), définissez l'**URI de redirection autorisée** exactement comme suit :

`{API_PUBLIC_URL}/oauth/callback`

- `API_PUBLIC_URL` est l'URL publique de base de l'API Rune (sans barre oblique finale), telle que définie dans l'environnement de l'API (voir `services/api/.env.example`).
- Si vous utilisez la variable optionnelle `OAUTH_REDIRECT_URI`, enregistrez **cette** URL complète à la place.

Exemples :

- API locale sur le port 8000 : `http://127.0.0.1:8000/oauth/callback`
- API derrière nginx sous `/api` : `https://your-host.example.com/api/oauth/callback`

Un écart entre cette URL et celle enregistrée chez le fournisseur est la cause la plus fréquente des erreurs `redirect_uri_mismatch`.

### Portées (scopes)

Saisissez les scopes sous forme d'une **seule chaîne séparée par des espaces** sur l'identifiant (par exemple Google Calendar en lecture seule). Si vous élargissez les scopes plus tard, mettez à jour l'identifiant et relancez **Connect** afin que le fournisseur émette des jetons pour le nouvel ensemble de scopes.

### Après la connexion

Définissez `OAUTH_FRONTEND_SUCCESS_URL` et `OAUTH_FRONTEND_ERROR_URL` vers des pages de votre application web (par défaut `/create/credentials`). L'API y redirige avec `?oauth=success` ou `?oauth=error&reason=...`.

Lors de l'exécution des workflows, les identifiants OAuth2 sont résolus côté serveur : les jetons sont **rafraîchis à l'approche de leur expiration** (avec marge), et seul un **sous-ensemble sûr** (`access_token`, `token_type`) est transmis au worker pour l'en-tête `Authorization` du nœud HTTP.
