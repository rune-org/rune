# Anmeldedaten verwalten

## OAuth2-Anmeldedaten (eigener OAuth-Client)

Rune verwendet den **Authorization-Code**-Grant: Ihr Browser besucht den Anbieter, Sie genehmigen den Zugriff, und die **API** tauscht den einmaligen `code` mit Ihrem **Client Secret** gegen `access_token` / `refresh_token` (der Browser sieht das Secret nie).

### Zu registrierende Redirect-URI

Tragen Sie in den OAuth-Client-Einstellungen Ihres Anbieters (z. B. Google Cloud Console → APIs & Services → Credentials → OAuth 2.0 Client) als **autorisierte Redirect-URI** exakt Folgendes ein:

`{API_PUBLIC_URL}/oauth/callback`

- `API_PUBLIC_URL` ist die öffentliche Basis-URL der Rune-API (ohne abschließenden Schrägstrich), wie in der API-Umgebung gesetzt (siehe `services/api/.env.example`).
- Wenn Sie die optionale Variable `OAUTH_REDIRECT_URI` verwenden, registrieren Sie stattdessen **diese** vollständige URL.

Beispiele:

- Lokale API auf Port 8000: `http://127.0.0.1:8000/oauth/callback`
- API hinter nginx unter `/api`: `https://your-host.example.com/api/oauth/callback`

Eine Abweichung zwischen dieser URL und der beim Anbieter hinterlegten ist die häufigste Ursache für `redirect_uri_mismatch`-Fehler.

### Scopes

Geben Sie Scopes als **eine einzelne, durch Leerzeichen getrennte** Zeichenkette auf der Anmeldedaten-Ressource ein (z. B. Google Calendar nur lesend). Wenn Sie die Scopes später erweitern, aktualisieren Sie die Anmeldedaten und führen Sie **Connect** erneut aus, damit der Anbieter Tokens für den neuen Scope-Satz ausstellt.

### Nach dem Verbinden

Setzen Sie `OAUTH_FRONTEND_SUCCESS_URL` und `OAUTH_FRONTEND_ERROR_URL` auf Seiten Ihrer Web-App (Standard: `/create/credentials`). Die API leitet dorthin mit `?oauth=success` oder `?oauth=error&reason=...` weiter.

Workflow-Ausführungen lösen OAuth2-Anmeldedaten auf dem Server auf: Tokens werden **bei nahender Ablaufzeit erneuert** (mit Toleranz), und nur eine **sichere Teilmenge** (`access_token`, `token_type`) wird für den `Authorization`-Header des HTTP-Knotens an den Worker gesendet.
