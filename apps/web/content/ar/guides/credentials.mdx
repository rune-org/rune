# إدارة بيانات الاعتماد

## بيانات اعتماد OAuth2 (استخدم عميل OAuth الخاص بك)

يستخدم Rune تدفق **رمز التفويض (authorization code)**: يزور متصفحك مزوّد الخدمة، وتوافق على منح الوصول، ثم تقوم **واجهة البرمجة (API)** باستبدال رمز `code` ذي الاستخدام الواحد بـ `access_token` / `refresh_token` باستخدام **سر العميل (client secret)** الخاص بك (المتصفح لا يرى السر أبداً).

### عنوان إعادة التوجيه الواجب تسجيله

في إعدادات عميل OAuth لدى مزوّدك (مثلاً Google Cloud Console → APIs & Services → Credentials → OAuth 2.0 Client)، عيّن **عنوان إعادة التوجيه المصرّح به** بالضبط كما يلي:

`{API_PUBLIC_URL}/oauth/callback`

- `API_PUBLIC_URL` هو عنوان URL العام الأساسي لواجهة برمجة Rune (بدون شرطة مائلة في النهاية)، كما هو محدد في بيئة الـ API (انظر `services/api/.env.example`).
- إذا كنت تستخدم المتغير الاختياري `OAUTH_REDIRECT_URI`، فسجّل **ذلك** العنوان الكامل بدلاً منه.

أمثلة:

- API محلي على المنفذ 8000: `http://127.0.0.1:8000/oauth/callback`
- API خلف nginx على المسار `/api`: `https://your-host.example.com/api/oauth/callback`

عدم تطابق هذا العنوان مع المسجّل لدى المزوّد هو السبب الأكثر شيوعاً لأخطاء `redirect_uri_mismatch`.

### النطاقات (Scopes)

أدخل النطاقات كسلسلة **واحدة مفصولة بمسافات** على بيانات الاعتماد (مثلاً تقويم Google للقراءة فقط). إذا وسّعت النطاقات لاحقاً، فحدّث بيانات الاعتماد وشغّل **Connect** مجدداً ليُصدر المزوّد رموزاً للنطاقات الجديدة.

### بعد الاتصال

عيّن `OAUTH_FRONTEND_SUCCESS_URL` و`OAUTH_FRONTEND_ERROR_URL` إلى صفحات في تطبيق الويب الخاص بك (القيم الافتراضية تشير إلى `/create/credentials`). تعيد الـ API التوجيه إليها مع `?oauth=success` أو `?oauth=error&reason=...`.

تُحلّ بيانات اعتماد OAuth2 أثناء تشغيل سير العمل على الخادم: تُجدَّد الرموز **عند اقتراب انتهاء صلاحيتها** (مع هامش زمني)، ولا يُرسل إلى العامل (worker) سوى **مجموعة فرعية آمنة** (`access_token`, `token_type`) لاستخدامها في ترويسة `Authorization` الخاصة بعقدة HTTP.
