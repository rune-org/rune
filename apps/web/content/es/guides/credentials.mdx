# Gestión de credenciales

## Credenciales OAuth2 (trae tu propio cliente OAuth)

Rune utiliza el flujo **authorization code**: tu navegador visita el proveedor, apruebas el acceso y la **API** intercambia el `code` de un solo uso por `access_token` / `refresh_token` usando tu **client secret** (el navegador nunca ve el secreto).

### URI de redirección a registrar

En la configuración del cliente OAuth de tu proveedor (por ejemplo Google Cloud Console → APIs & Services → Credentials → OAuth 2.0 Client), establece la **URI de redirección autorizada** exactamente como:

`{API_PUBLIC_URL}/oauth/callback`

- `API_PUBLIC_URL` es la URL base pública de la API de Rune (sin barra final), tal como se define en el entorno de la API (ver `services/api/.env.example`).
- Si usas la variable opcional `OAUTH_REDIRECT_URI`, registra **esa** URL completa en su lugar.

Ejemplos:

- API local en el puerto 8000: `http://127.0.0.1:8000/oauth/callback`
- API detrás de nginx en `/api`: `https://your-host.example.com/api/oauth/callback`

Una discrepancia entre esta URL y la registrada en el proveedor es la causa más común de los errores `redirect_uri_mismatch`.

### Ámbitos (scopes)

Introduce los scopes como **una sola cadena separada por espacios** en la credencial (por ejemplo Google Calendar de solo lectura). Si amplías los scopes más adelante, actualiza la credencial y ejecuta **Connect** de nuevo para que el proveedor emita tokens con el nuevo conjunto de scopes.

### Después de conectar

Configura `OAUTH_FRONTEND_SUCCESS_URL` y `OAUTH_FRONTEND_ERROR_URL` hacia páginas de tu aplicación web (por defecto apuntan a `/create/credentials`). La API redirige allí con `?oauth=success` o `?oauth=error&reason=...`.

Las ejecuciones de workflows resuelven las credenciales OAuth2 en el servidor: los tokens se **refrescan cuando están cerca de expirar** (con margen), y solo un **subconjunto seguro** (`access_token`, `token_type`) se envía al worker para la cabecera `Authorization` del nodo HTTP.
